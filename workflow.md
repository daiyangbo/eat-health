# Claude Code 通用工作流程

> **用途**：适用于所有项目的通用工作流程和规则
> **版本**：v1.0
> **创建日期**：2026-06-11

---

## 一、回复格式规则

每次回复必须以固定前缀开头（如"报告老大"），保持一致性。

---

## 二、需求澄清规则

当用户提出的需求过于宽泛或不明确时，应主动反问以确认具体要求。

### 示例

| 用户说 | 应反问 |
|--------|--------|
| "写个网站" | 基于什么技术栈、什么框架、什么风格 |
| "做个功能" | 具体实现方式、使用场景、技术要求 |
| "优化代码" | 优化方向（性能/可读性/结构） |

---

## 三、确认清单

在开始工作前，确保明确以下要素：

- [ ] 具体使用的技术和工具
- [ ] 具体的实现目标
- [ ] 具体的输出格式
- [ ] 具体的保存要求

---

## 四、工作流程规则

**核心原则：一步一步完成任务，先分析再做**

### 流程

```
收到任务
    │
    ▼
拆解为多个步骤
    │
    ▼
┌─────────────────────────────────────┐
│ 对每个步骤：                          │
│   1. 分析（调研、理解、规划）           │
│   2. 执行（编码、实现）                │
│   3. 汇报进度                         │
│   4. 进入下一步                        │
└─────────────────────────────────────┘
    │
    ▼
完成任务
```

### 要求

- 使用 TodoWrite 工具记录任务进度
- 完成一个步骤后，汇报进度，再进入下一步
- 遇到问题及时反馈，不要盲目推进

---

## 五、对话打包规则

当用户说"打包对话"时，执行以下操作：

### 步骤

1. **创建对话总览文件**（如不存在）
   - 文件名：`对话总览.md`
   - 内容：项目进度、对话记录表格、文件结构说明

2. **创建对话记录文件**
   - 文件名：`对话记录/第X次对话记录.md`
   - 内容：本次对话的所有有用信息

3. **更新对话总览**
   - 在对话记录表格中添加本次对话记录
   - 更新项目进度

---

## 六、新对话启动规则

每次开新对话时，必须先读取以下内容：

### 必读内容

1. **对话总览**
   - 读取 `对话总览.md` 了解项目整体进度和历史对话内容
   - 确定当前阶段

2. **当前阶段项目文件夹**
   - 根据 `对话总览.md` 中的当前阶段，读取对应的项目文件夹内容
   - 如当前阶段是 MVP，则读取 MVP 文件夹中的工作流程和待办事项

3. **学习记录**（如有）
   - 读取学习记录文件夹中的相关文件

### 读取顺序

```
1. 对话总览.md → 了解整体进度
2. 当前阶段文件夹/ → 了解当前任务
3. 学习记录/ → 了解背景知识
```

---

## 七、参考资料规则

- 在设计功能、算法、用户体验时，可以适当参考竞品分析报告
- 参考时注明引用来源，但不要完全照搬，需根据实际场景进行调整
- 竞品分析报告应包含：功能分析、技术栈、优点、可借鉴之处

---

## 八、数据来源规则

### 要求

- 给出任何数据、事实、统计信息时，必须标注数据来源
- 如果是推测或假设，必须明确标注为"推测"或"假设"

### 来源格式

```
来源：[网站/文档名称](URL)
```

### 示例

```
来源：[eatthismuch.com](https://www.eatthismuch.com/)
来源：[Qwen2.5文档](https://huggingface.co/Qwen/Qwen2.5-1.5B)
推测：基于行业惯例推断
```

## 十、版本演进策略

### 核心原则

```
功能固定 + 数据扩展

功能：一次性定好，后续不轻易改动
数据：分批扩展，逐步完善
```

### 版本定义

| 版本 | 目标 | 数据范围 |
|------|------|----------|
| MVP | 验证核心能力 | 最小可用数据 |
| V1.0 | 第一个场景上线 | 单场景数据 |
| V1.5 | 扩展场景 | 多场景数据 |
| V2.0 | 完整覆盖 | 全量数据 |
