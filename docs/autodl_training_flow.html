第三阶段：环境配置
4
安装依赖包并设置镜像
在 AutoDL 实例上安装项目所需的 Python 包，并设置 HuggingFace 镜像加速模型下载。
# 进入项目目录
cd /root/eat-health

# 清理缓存（释放 / 目录空间）
pip cache purge

# 安装依赖到 /root 目录（空间更充足）
pip install -r requirements.txt --target /root/pip_packages --no-cache-dir

# 设置 Python 路径
export PYTHONPATH=/root/pip_packages:$PYTHONPATH

# 设置 HuggingFace 镜像（重要！加速模型下载）
export HF_ENDPOINT=https://hf-mirror.com

为什么要安装到 /root 目录？

  • AutoDL 的 / 目录只有 30GB，空间紧张
  • /root 目录有 98GB，空间充足
  • PyTorch + CUDA 依赖约 2-3GB，需要足够空间

为什么要用 --no-cache-dir？

  • 不缓存可以节省 2-3GB 磁盘空间
  • 避免把 / 目录撑满
  • 对后续训练没有影响

为什么要设置 HuggingFace 镜像？

  • Qwen2.5-7B 模型约 14GB，直接下载很慢
  • 使用镜像可以加速 5-10 倍
  • 这是国内用户的标准做法
⏱️ 预计 10 分钟 ⚠️ 确保 / 目录有至少 5GB 可用空间
⬇️
5
下载基座模型
下载 Qwen2.5-7B 模型到本地，用于后续微调。
# 设置 HuggingFace 镜像（国内加速）
export HF_ENDPOINT=https://hf-mirror.com

# 使用下载脚本（推荐）
python download_model.py

下载脚本功能：

  • 自动检查模型是否已下载
  • 支持重试机制（最多 3 次）
  • 显示下载进度和模型信息
  • 保存到 ./models/base/qwen2.5-7b/
⏱️ 预计 10-15 分钟 📦 模型大小约 14GB
⬇️
6
安装 LLaMA-Factory
安装 LLaMA-Factory 微调框架，用于 SFT 训练。需要使用兼容 Python 3.10 的版本。
# 清理旧版本（如果存在）
rm -rf /root/pip_packages/llamafactory
rm -rf /root/eat-health/LLaMA-Factory

# 克隆兼容版本（v0.9.3，支持 Python 3.10）
cd /root/eat-health
git clone --branch v0.9.3 https://ghfast.top/https://github.com/hiyouga/LLaMA-Factory.git

# 安装到系统目录
cd LLaMA-Factory
pip install -e . --no-cache-dir

为什么要用 v0.9.3 版本？

  • 最新版 LLaMA-Factory 需要 Python 3.11+
  • AutoDL 默认是 Python 3.10，不兼容
  • v0.9.3 支持 Python 3.10，稳定可靠

为什么要用镜像站？

  • GitHub 直连很慢，容易超时
  • ghproxy.net 是 GitHub 镜像站，速度快
⏱️ 预计 5-10 分钟

第五阶段：模型训练
8
执行 SFT 微调
使用 LLaMA-Factory 进行监督微调训练。
# 进入项目根目录
cd /root/eat-health

# 运行训练（使用 Python 模块方式）
python -m llamafactory.cli train /root/eat-health/train_config.yaml

为什么要用 python -m 方式？

  • llamafactory-cli 命令可能找不到
  • 使用 Python 模块方式更可靠
  • 需要在项目根目录运行

训练参数详解（已优化）：

参数 值 说明
model_name_or_path ./models/base/qwen2.5-7b 基座模型路径
finetuning_type lora 使用 LoRA 微调（节省显存）
num_train_epochs 10 训练轮数（从3增加到10，让模型充分学习）
per_device_train_batch_size 1 每个设备的批次大小（RTX 4090 用 1）
gradient_accumulation_steps 16 梯度累积步数（有效批次 = 1×16 = 16）
learning_rate 2e-6 学习率（7B模型使用更小的学习率）
lr_scheduler_type cosine 学习率调度器
warmup_steps 300 预热步数（7B模型需要更多预热）
bf16 true 使用 bfloat16 精度（节省显存）
save_steps 300 每 300 步保存一次 checkpoint（从500调整）
logging_steps 10 每 10 步记录一次日志

为什么使用 LoRA？

  • RTX 4090 显存 24GB，全参数微调会 OOM
  • LoRA 只训练约 0.8% 参数（56M vs 7B）
  • 训练速度快，4-5 分钟完成
  • 效果接近全参数微调

为什么批次大小设为 1？

  • 避免 CUDA 显存不足错误
  • 通过梯度累积（16 步）保持有效批次大小
  • 有效批次 = 1 × 16 = 16
⏱️ 预计 2-3 小时 💰 约 ¥50-75

🔧 常见问题及解决方案

问题 1：CUDA 内存不足

torch.OutOfMemoryError: CUDA out of memory

解决方案：
1. 使用 LoRA 微调：finetuning_type: lora（推荐，只训练约 0.8% 参数）
2. 减小批次大小：per_device_train_batch_size: 1
3. 增加梯度累积：gradient_accumulation_steps: 16
4. 使用梯度检查点：添加 gradient_checkpointing: true

问题 2：GitHub 克隆超时

fatal: unable to access: Connection timed out

解决方案：
1. 开启学术加速：source /etc/network_turbo
2. 使用镜像站克隆：git clone https://ghfast.top/https://github.com/daiyangbo/eat-health.git
3. 如果还是慢，等几分钟再试

问题 3：磁盘空间不足（No space left on device）

OSError: [Errno 28] No space left on device

原因：AutoDL 的 / 目录只有 30GB，PyTorch + CUDA 依赖约 2-3GB，加上缓存可能撑满。

解决方案：
1. 扩展存储：在 AutoDL 控制台扩展到 50GB 以上
2. 清理缓存：pip cache purge
3. 安装到 /root 目录：pip install -r requirements.txt --target /root/pip_packages --no-cache-dir
4. 设置 Python 路径：export PYTHONPATH=/root/pip_packages:$PYTHONPATH

问题 4：模型下载失败

ConnectionError 或 TimeoutError

解决方案：
1. 设置 HuggingFace 镜像：export HF_ENDPOINT=https://hf-mirror.com
2. 确保已开启学术加速：source /etc/network_turbo
3. 使用代理
4. 手动下载模型并上传到 ./models/base/qwen2.5-7b/

问题 5：LLaMA-Factory 安装失败

ERROR: Package 'llamafactory' requires a different Python: 3.10.8 not in '>=3.11.0'

原因：最新版 LLaMA-Factory 需要 Python 3.11+，但 AutoDL 默认是 Python 3.10。

解决方案：
1. 使用兼容版本：git clone --branch v0.9.3 https://ghfast.top/https://github.com/hiyouga/LLaMA-Factory.git
2. 清理旧版本：rm -rf /root/pip_packages/llamafactory
3. 安装到系统目录：pip install -e . --no-cache-dir

问题 6：transformers 版本冲突

ImportError: transformers>=4.45.0,<=4.52.4 is required

解决方案：
1. 跳过版本检查：export DISABLE_VERSION_CHECK=1
2. 或降级 transformers：pip install transformers==4.52.4

问题 7：训练速度慢

解决方案：
1. 确认使用了 GPU（检查 nvidia-smi）
2. 增加 dataloader_num_workers
3. 使用混合精度训练（已启用 bf16）

问题 8：训练 Loss 不下降

解决方案：
1. 检查数据格式是否正确
2. 调整学习率（尝试 2e-5 或 5e-6）
3. 增加训练轮数
4. 检查数据质量

问题 9：模型输出有重复、乱码或幻觉

creampie creampie ... 或 桤杣user ... 或 无关话题

原因：训练轮次不足（默认3轮太少），模型没学会"何时停止"或"输出正确格式"。

解决方案：
1. 增加训练轮次（推荐）：将 num_train_epochs 从 3 改为 10 或 15
2. 降低学习率：将 learning_rate 从 1e-5 改为 5e-6，防止过拟合
3. 增加预热步数：将 warmup_steps 从 100 改为 200
4. 调整保存频率：将 save_steps 从 500 改为 300
5. 增加数据量：用 GPT-4 生成更多高质量训练数据（1000+条）
6. 使用更大模型：已升级到 7B 模型（效果提升显著）

问题 9：显存占用过高

解决方案：
1. 使用 nvidia-smi 监控显存
2. 减小 max_length（如果设置了的话）
3. 使用梯度检查点：gradient_checkpointing: true

问题 10：llamafactory-cli 命令找不到

/bin/sh: 1: llamafactory-cli: not found

解决方案：
使用 Python 模块方式运行：python -m llamafactory.cli train /root/eat-health/train_config.yaml

问题 11：训练配置修改不生效

修改 train_config.yaml 后，重新训练仍显示 Num Epochs = 3

原因：可能是代码缓存问题，或者 train.py 脚本有自己的默认配置覆盖了配置文件。

解决方案：
1. 使用 sed 命令直接修改配置文件：sed -i 's/num_train_epochs: 3/num_train_epochs: 10/' train_config.yaml
2. 修改后用 cat train_config.yaml | grep num_train_epochs 确认修改成功
3. 清除旧模型：rm -rf models/finetuned/*
4. 重新训练：python train.py

问题 12：模型输出有重复、乱码或幻觉

creampie creampie ... 或 桤杣user ... 或 无关话题

现象：
- 重复生成（Degeneration）：出现 creampie creampie ... 无限循环
- 对话模板泄露：出现 桤杣user / 桤杣assistant 对话格式
- 幻觉（Hallucination）：输出跳转到完全无关话题（如树皮、工作描述）

根本原因：
1. 训练轮次不足（默认3轮太少），模型没学会"何时停止"
2. 原模型容量小（已从1.5B升级到7B）
3. 训练数据量不足（480条）

解决方案（已验证有效）：
1. 增加训练轮次（推荐）：将 num_train_epochs 从 3 改为 10 或 15
2. 降低学习率：将 learning_rate 从 1e-5 改为 5e-6，防止过拟合
3. 增加预热步数：将 warmup_steps 从 100 改为 200
4. 调整保存频率：将 save_steps 从 500 改为 300
5. 优化生成参数：降低 temperature(0.3)、增加 repetition_penalty(1.3)、添加 no_repeat_ngram_size(3)
6. 统一输入格式：确保测试输入格式与训练数据格式一致
7. 增加数据量：用 GPT-4 生成更多高质量训练数据（1000+条）
8. 使用更大模型：已升级到 7B 模型（效果提升显著）

问题 13：训练数据格式与测试输入不一致

训练数据：蛋白质101g，脂肪88g，碳水250g
测试输入：请帮我搭配一餐，营养目标：蛋白质150g，脂肪60g，碳水化合物200g

影响：模型无法正确理解输入，输出格式不稳定。

解决方案：
1. 修改 test_model.py 中的测试输入格式，与训练数据保持一致
2. 使用简洁格式：蛋白质150g，脂肪60g，碳水200g
3. 简化 instruction，不要添加多余描述

问题 14：食材名称是英文（USDA格式）

Pork, fresh, leg (ham), shank half, separable lean only, raw 300g

影响：用户体验差，不符合学校食堂场景。

解决方案（待实施）：
1. 建立英文-中文食材名称映射表
2. 重新生成训练数据，使用中文食材名称
3. 或者在后处理阶段将英文名称转换为中文
