## 一、PDF内容概览

### 1. GUI智能体概述
- AGI五阶段：Chatbots → Reasoners → Agents（当前阶段）→ Innovators → Organizations
- 核心转变：从"内容智能"到"行为智能"，语言→行动
- GUI智能体定义：主动规划决策，模拟人类行为（点击、滑动、输入），完成信息获取和功能执行的智能体
- GUI vs API：API快捷稳定但笨重，GUI灵活自然但脆弱，不存在完美方案

### 2. 技术分支
- 基于闭源模型：Mobile-Agent-v2（规划+决策+反思三智能体）、AppAgentX（学习捷径节点）
- 基于开源模型：OS-Atlas、UI-TARS、CogAgent、Aguvis，可用预训练/指令微调/强化学习构建

### 3. GUI智能体部署
- 安卓真机：ADB命令（`adb shell input tap <x> <y>`）
- 安卓虚拟机：AVD
- 电脑：pyautogui库（`pyautogui.click(x, y)`）

### 4. 三种场景方案
- 静态场景 → 监督微调（SFT）：输入截图+指令，输出动作。代表：OS-Atlas、AutoGLM
- 动态场景 → 强化学习：适应环境变化（网页布局、软件更新）。代表：DigiRL、DistRL、GRPO（UI-R1、GUI-R1）
- 复杂场景 → 调度系统：拆解复合任务（拼接型/传递型/深度分析型）。代表：UI-NEXUS

### 5. 安全与可靠性
- 安全问题：注入攻击（恶意提示、恶意图像补丁）、鲁棒性不足、安全对齐隐患
- 可靠性问题：OOD问题（静态训练→动态真实环境）、过度执行、回退机制、模糊任务

## 二、本教程实操：OS-Kairos GUI智能体构建

### 1. 项目概述

#### 实验目标
基于开源模型Qwen2-VL-7B，依托OS-Kairos数据集，通过SFT构建自适应人机交互GUI智能体。

#### 核心思路
> 用带标注的截图-动作数据对Qwen2-VL-7B做SFT，让模型学会"看屏幕截图→输出操作指令+置信度分数"，通过置信度分数实现人机自适应交互。

#### 技术栈
- 基座模型：Qwen2-VL-7B-Instruct（7B参数多模态视觉语言模型）
- 数据集：OS-Kairos（手机GUI操作标注数据）
- 训练框架：LLaMA-Factory（支持Qwen2-VL全量微调）
- 硬件要求：至少3张A100 80GB

### 3. 核心知识点详解

#### Q1: Prompt怎么设计？

整个prompt是固定模板+3个变量：

```
┌─────────────────────────────────────────────────────┐
│              固定部分（每条数据都一样）                  │
├─────────────────────────────────────────────────────┤
│ ① 角色设定：Executable Language Grounding mode       │
│ ② 动作空间定义：7种动作的格式说明                      │
│ ③ 评分机制说明：1-5分含义                             │
├─────────────────────────────────────────────────────┤
│              可变部分（每条数据不同）                    │
├─────────────────────────────────────────────────────┤
│ ④ Final goal: {task}           ← 用户指令            │
│ ⑤ previous actions: {prev}     ← 已执行动作           │
│ ⑥ screenshot:<image>           ← 当前截图             │
├─────────────────────────────────────────────────────┤
│              输出格式约束                              │
├─────────────────────────────────────────────────────┤
│ action:                                             │
│ score:                                              │
└─────────────────────────────────────────────────────┘
```

设计原因：
- 先定义动作空间 → 让模型知道"我能做什么"，限制输出在合法范围内
- 再给评分机制 → 让模型学会自我评估，输出置信度而非盲目执行
- 最后给任务 → 在模型已知规则后，再告诉它"要做什么"
- 严格输出格式 → 方便后处理解析

#### Q2: 动作空间怎么设计？

一共7种动作，分两类：

**基础动作（3种）—— 带坐标参数：**
| 动作 | 格式 | 用途 |
|------|------|------|
| CLICK | `CLICK <point>[[x, y]]</point>` | 点击屏幕某位置 |
| TYPE | `TYPE [文本内容]` | 在输入框输入文字 |
| SCROLL | `SCROLL [UP/DOWN/LEFT/RIGHT]` | 滑动屏幕 |

**自定义动作（4种）—— 无参数：**
| 动作 | 用途 |
|------|------|
| PRESS_BACK | 返回上一页 |
| PRESS_HOME | 回到主屏幕 |
| ENTER | 按回车键 |
| IMPOSSIBLE | 声明任务无法完成（安全阀） |

设计思路：基础动作覆盖通用GUI操作（点击/输入/滑动），自定义动作覆盖系统级操作（返回/主页/回车/放弃），动作空间小而精，模型容易学会。

#### Q3: 置信度分数怎么用？

**训练时：** 分数是标注员给出的（1-5），作为监督信号的一部分

**推理时：** 模型自己输出分数，用于判断是否需要人类介入

```
score = 5  →  模型非常确定  →  自动执行
score = 4  →  模型比较确定  →  自动执行
score = 3  →  模型不太确定  →  可尝试，需观察结果
score = 2  →  模型很不确定  →  暂停，请求人类确认
score = 1  →  模型认为做不到 →  停止，人类接管
```

这就是"自适应人机交互"的核心：模型自己判断能力边界，能力范围内自动执行，超出范围时及时求助人类。

#### Q4: 数据预处理做了什么？

将OS-Kairos原始JSON转为LLaMA-Factory要求的sharegpt格式：

```
原始数据（OS-Kairos格式）          →    转换后（sharegpt格式）
┌──────────────────────┐           ┌──────────────────────────────────┐
│ task: "打开网易云..."   │           │ messages: [                      │
│ image_path: "xxx.png" │    →      │   {role: "user", content: "      │
│ osatlas_action: "..." │           │     完整prompt（含动作定义+任务）"},│
│ score: 5              │           │   {role: "assistant", content:    │
└──────────────────────┘           │     "action: ...\nscore: ..."}]   │
                                   │ images: ["xxx.png"]               │
                                   └──────────────────────────────────┘
```

每个样本被包装成一轮对话：user提供截图+完整prompt，assistant输出动作+分数。模型通过SFT学会这种对话模式。

#### Q5: 训练配置关键参数

| 参数 | 值 | 含义 |
|------|-----|------|
| finetuning_type | full | 全量微调（不是LoRA） |
| cutoff_len | 4096 | 最大序列长度 |
| per_device_train_batch_size | 2 | 每卡batch=2 |
| gradient_accumulation_steps | 2 | 梯度累积2步 |
| 有效batch_size | 2×8×2=32 | 8卡×每卡2×累积2 |
| learning_rate | 1.0e-5 | 学习率 |
| num_train_epochs | 5.0 | 训练5轮 |
| lr_scheduler_type | cosine | 余弦退火 |
| val_size | 0.1 | 10%数据做验证集 |

### 4. 与前序章节的对比

#### Ch1 vs Ch8 vs Ch9

| | Ch1 大模型微调 | Ch8 多模态 | Ch9 GUI Agent |
|--|---------------|-----------|---------------|
| 基座模型 | 纯语言模型 | NExT-GPT | Qwen2-VL-7B |
| 输入 | 文本 | 图像/视频/音频+文本 | 截图+任务指令 |
| 输出 | 文本回答 | 文本/图像/音频/视频 | 动作指令+置信度分数 |
| 训练框架 | Transformers Trainer | 自定义三阶段代码 | LLaMA-Factory |
| GPU需求 | 单卡40GB | 多卡 | 3×A100 80GB |
| 写代码量 | 多（自己写训练代码） | 多（理解多组件架构） | 少（改配置文件+一行命令） |

#### 本质关系
```
Ch1 是底层基础：SFT的原理（怎么微调一个模型）
  ↓
Ch8 是能力扩展：给模型加上"眼睛"（多模态理解）
  ↓
Ch9 是任务落地：用多模态模型解决具体问题（GUI操作）

Ch9 = Ch1的SFT方法 + Ch8的多模态架构 + GUI任务定义
```

#### 与YOLO的对比
- 相似点：都是"看图→输出坐标+置信度"
- 核心差异：YOLO是纯视觉检测（看到什么），GUI Agent是多模态+语义理解+决策（要做什么）
- GUI Agent = YOLO的视觉定位能力 + LLM的语言理解能力 + 决策能力的融合
