## 一、PDF内容概览

### 1. 为什么需要多模态

- 语言是承载智能的关键模态（规模定律 + 涌现现象证明了这一点）
- 但真实世界是多模态的：图像、文本、音频、视频
- 目标：让 LLM 像人一样感知多种模态 → 更接近 AGI

### 2. MLLM 的类型分类

**按输入模态：**
- 文本 + 图像（最常见：LLaVA、GPT-4V）
- 文本 + 视频
- 文本 + 音频
- 任意模态组合（如 NExT-GPT）

**按输出模态：**
- 只输出文本 → 理解型（看图说话、视频问答）
- 输出文本 + 图像/音频/视频 → 生成型（文生图、文生视频）

**按功能：**
- 多模态理解：看图说话、视频问答
- 多模态生成：文字生图、文字生视频
- 任意到任意：输入任意模态 → 输出任意模态（NExT-GPT）

### 3. 两种核心架构

**架构一：LLM as Task Scheduler（调度器架构）**
- 结构：LLM + 多个独立功能模块
- 通信：LLM 输出纯文本命令 → 各模块执行
- 特点：模块间无直接交互，LLM 是"指挥官"
- 比喻：项目经理（LLM）用文字指令分配任务给各部门
- 缺点：延迟高、信息损失（文字描述不如直接传向量）

**架构二：LLM as Joint Part of System（编码器-LLM-解码器）**
- 结构：编码器 → LLM → 解码器
- 通信：编码器输出向量直接送入 LLM
- 特点：LLM 是核心决策者，直接感知多模态信息
- 比喻：大脑（LLM）直接接收眼睛/耳朵的神经信号
- 优势：信息无损、延迟低、端到端训练
- 代表：LLaVA、NExT-GPT、GPT-4V/4o

## 二、NExT-GPT 实战详解

### 1. 整体架构

NExT-GPT 是首个"任意到任意模态"的 MLLM，支持文字/图像/视频/音频四种模态的任意组合输入输出。

```
输入端                    中间                     输出端
┌─────────────┐    ┌──────────────┐    ┌─────────────────────┐
│  ImageBind  │    │    Vicuna    │    │ Stable Diffusion    │
│ (统一编码器) │───→│   (LLM)     │───→│ (生成图像)           │
│ 图像/视频/音频│    │  理解+决策   │    │ AudioLDM (生成音频)  │
└─────────────┘    └──────────────┘    │ ZeroScope (生成视频) │
       ↑                   ↑           └─────────────────────┘
  输入投影层           输出投影层
  (翻译官)             (翻译官)
```

**输入端：ImageBind（统一编码器）**
- 一个模型编码图像/视频/音频，输出统一的多模态嵌入向量
- 以前每个模态用不同编码器（CLIP处理图像、Whisper处理音频...），ImageBind 用一个模型搞定

**中间：Vicuna（基于 LLaMA 的 LLM）**
- 接收多模态嵌入 → 理解 + 决策 → 输出文本 + 信号 Token

**输出端：三个扩散模型**
- Stable Diffusion → 生成图像
- AudioLDM → 生成音频
- ZeroScope → 生成视频

### 2. 关键组件：投影层（Projection Layer）

投影层是 NExT-GPT 的核心创新之一，作用是桥接不同模态的表征空间。

**输入投影层：**
- 编码器输出 → LLM 可理解的向量
- 把"外语"翻译成 LLM 的"母语"

**输出投影层：**
- LLM 输出 → 扩散模型可理解的向量
- 把 LLM 的"母语"翻译成扩散模型的"外语"

**为什么需要投影层？**
不同模型的"语言"不同。ImageBind 输出的向量和 Vicuna 理解的向量不在同一个空间。投影层就像翻译官，把一种模态的"方言"翻译成另一种模态的"普通话"。

### 3. 三阶段训练策略

| 阶段 | 目标 | 训练什么 | 冻结什么 | 数据 | 类比 |
|------|------|---------|---------|------|------|
| Stage 1 编码端对齐 | 让 LLM "看懂"图像/视频/音频 | 输入投影层 | ImageBind、LLM、输出投影层 | CC3M、WebVid、AudioCap | 教 LLM 学会"看"和"听" |
| Stage 2 解码端对齐 | 让扩散模型"听懂"LLM 的指令 | 输出投影层 | ImageBind、LLM、输入投影层 | T-X 对数据的 caption 嵌入（预计算） | 教扩散模型学会"听从指挥" |
| Stage 3 指令微调 | 让整个系统理解复杂指令并协调输出 | LoRA 调 LLM + 输入投影层 + 输出投影层 | ImageBind、扩散模型主体 | Alpaca、LLaVA、VideoChat、MosIT | 教会 LLM "多模态协调能力" |

**为什么分三阶段而不是一次性全训？**
- 类比盖楼：先装好窗户（Stage 1 对齐输入），再装好门（Stage 2 对齐输出），最后精装修（Stage 3 指令微调）
- 一次性全训参数太多，容易互相干扰，梯度会"打架"
- 分阶段训练，每阶段目标明确，收敛更快更稳定

### 4. 数据准备

**T-X 配对数据（用于 Stage 1 & 2）：**
- CC3M：300万文本-图像对
- WebVid：文本-视频对
- AudioCap：文本-音频对

**指令微调数据（用于 Stage 3）：**
- Alpaca：纯文本指令（保持 LLM 的文本对话能力）
- LLaVA：视觉指令（教模型理解图文指令）
- VideoChat：视频指令（教模型理解视频指令）
- MosIT：模态切换指令（教模型在不同模态间切换）

**嵌入预计算：**
- 用扩散模型的文本编码器预计算 caption 嵌入
- 目的：训练时直接读取，节省大量时间和显存
- 命令：`python process_embeddings.py cc3m.json image ../embed/ stable-diffusion-v1-5`

## 三、核心概念对比

### 编码器 vs 解码器 vs LLM

| 组件 | 作用 | 比喻 |
|------|------|------|
| 编码器 | 把多模态信号 → 向量（理解） | 眼睛/耳朵 |
| LLM | 接收向量 → 决策 + 生成文本（思考） | 大脑 |
| 解码器 | 接收向量 → 生成多模态内容（生成） | 嘴巴/手 |

### 投影层 vs 微调

| | 投影层 | 微调（LoRA） |
|--|--------|-------------|
| 是什么 | 轻量级线性层 | 修改 LLM 本身的能力 |
| 作用 | 只对齐表征空间（翻译官） | 改造大脑 |
| 参数量 | 很少 | LoRA 也很少（0.1%~1%） |
| 使用阶段 | Stage 1 & 2 | Stage 3 |

### 冻结 vs 训练

| | 冻结 | 训练 |
|--|------|------|
| 含义 | 保持预训练权重不变 | 更新权重以适配新任务 |
| 优点 | 省资源、保留已有能力 | 能适配新任务 |
| 使用策略 | 已经训好的模型（ImageBind、扩散模型） | 需要适配的组件（投影层、LoRA） |

### 推理 vs 训练

| | 推理（Demo） | 训练 |
|--|-------------|------|
| 目的 | 测试模型效果 | 生产模型权重 |
| 做什么 | 加载已有权重 → 启动网页 | 用数据更新参数 → 产出权重 |
| 命令 | `bash scripts/app.sh` | `bash scripts/train.sh --stage 1/2/3` |
| 结果 | 网页交互界面 | 保存到 `delta_ckpt/` 的权重文件 |

## 五、遇到的问题及解决

### 问题1：scipy 版本不兼容
**原因：** `scipy==1.14.2` 要求 Python ≥ 3.11，但环境是 Python 3.10
**解决：** `sed -i 's/scipy==1.14.2/scipy==1.13.1/' requirements.txt`

### 问题2：torchvision 版本不匹配
**原因：** requirements.txt 中的 `torchvision==0.17.1+cu121` 是 CUDA 12.1 版本，但环境是 CUDA 11.8
**解决：** `sed -i 's/torchvision==0.17.1+cu121/torchvision==0.16.2/' requirements.txt`

### 问题3：/tmp 目录不存在
**原因：** AutoDL 容器的临时目录被清理
**解决：** `mkdir -p /tmp && chmod 1777 /tmp`

### 问题4：代码目录结构与文档不一致
**原因：** 实际代码在 `NExT-GPT-Lagacy/code/` 而非文档说的 `code/`
**解决：** 使用实际路径 `/root/NExT-GPT/NExT-GPT-Lagacy/code/`

### 问题5：模型权重路径不匹配
**原因：** 下载的权重在 `/root/NExT-GPT/ckpt/` 但 Legacy 代码期望在 `NExT-GPT-Lagacy/ckpt/`
**解决：** `cp -r /root/NExT-GPT/ckpt/* /root/NExT-GPT/NExT-GPT-Lagacy/ckpt/`

## 八、重点答疑

### Q1: 为什么需要投影层？
不同模型的"语言"不同。ImageBind 输出的向量和 Vicuna 理解的向量不在同一个空间。投影层就像翻译官，把一种模态的"方言"翻译成另一种模态的"普通话"。

### Q2: 三阶段训练为什么不一次性全训？
类比盖楼：先装好窗户（Stage 1 对齐输入），再装好门（Stage 2 对齐输出），最后精装修（Stage 3 指令微调）。一次性全训参数太多，容易互相干扰，梯度会"打架"。

### Q3: ImageBind 为什么是"统一编码器"？
以前每个模态用不同编码器（CLIP 处理图像、Whisper 处理音频...），ImageBind 用一个模型就能编码图像/视频/音频，简化了架构。

### Q4: Demo 和训练的关系是什么？
Demo 不会训练任何东西，只是加载已有模型权重 + 启动 Gradio 网页进行推理测试。训练是独立的过程，产出的权重文件给 Demo 使用。类比：训练 = 工厂生产手机，Demo = 你拿到手机直接用。

### Q5: 为什么用 LoRA 而不是全量微调 LLM？
- 全量微调 7B 参数的 LLM 需要巨大显存和计算资源
- LoRA 只训练一小部分低秩矩阵（通常只有原参数的 0.1%~1%），效果接近全量微调
- 这是 Ch1（BERT 微调）中学过的同类技术

### Q6: 嵌入预计算的目的是什么？
训练时需要对比"Signal Token 的输出"和"caption 的表征"之间的距离。如果每次训练都实时跑一遍扩散模型的文本编码器，太慢了。提前算好存起来，训练时直接读取，节省大量时间和显存。

## 九、术语表

| 术语 | 全称 | 含义 |
|------|------|------|
| MLLM | Multimodal Large Language Model | 多模态大语言模型 |
| ImageBind | - | Meta 的统一多模态编码器，一个模型编码图像/视频/音频 |
| Vicuna | - | 基于 LLaMA 的开源 LLM |
| LoRA | Low-Rank Adaptation | 低秩适配，用少量参数微调大模型的技术 |
| 投影层 | Projection Layer | 桥接不同模态表征空间的线性层 |
| 信号 Token | Signal Token | LLM 输出的特殊 token，用于向解码器传递生成指令 |
| 扩散模型 | Diffusion Model | 通过逐步去噪生成图像/音频/视频的模型 |
| Stable Diffusion | - | 开源图像生成扩散模型 |
| AudioLDM | - | 开源音频生成扩散模型 |
| ZeroScope | - | 开源视频生成扩散模型 |
| DeepSpeed | - | 微软的分布式训练框架 |
| Gradio | - | 快速构建 ML 模型 Web UI 的 Python 库 |
| 对齐 | Alignment | 让不同模态的表征空间匹配的过程 |
| 指令微调 | Instruction Tuning | 用指令数据训练模型理解和执行指令 |
| 嵌入 | Embedding | 将数据转换为向量表示 |
| 预计算 | Pre-computation | 提前计算好结果，训练时直接读取 |
