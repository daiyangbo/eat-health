#### F1：营养目标输入

**是什么**：用户手动输入每天需要摄入的蛋白质、脂肪、碳水化合物的克数。

**为什么需要**：
- 这是最基础的输入方式，适合已经知道自己营养目标的用户
- 健身人群通常知道自己需要多少蛋白质
- 减脂人群通常知道自己需要控制多少碳水

**怎么工作**：
```
用户输入：
  请输入蛋白质（g）: 150
  请输入脂肪（g）: 60
  请输入碳水（g）: 200

系统接收这三个数值，作为后续推荐的目标。
```

**输入格式**：
| 参数 | 类型 | 范围 | 单位 |
|------|------|------|------|
| 蛋白质 | 整数 | 30-300 | 克(g) |
| 脂肪 | 整数 | 20-150 | 克(g) |
| 碳水 | 整数 | 50-500 | 克(g) |

**输出**：三个数值，传递给后续功能使用。

#### F2：交互式引导

**是什么**：通过一步步问答，帮助不知道自己营养目标的用户计算出目标值。

**为什么需要**：
- 很多学生不知道自己需要多少蛋白质/脂肪/碳水
- 直接输入克数太专业，普通人不懂
- 通过几个简单问题，系统自动计算，降低使用门槛

**怎么工作**：
```
Step 1: 问用户目标（减脂/维持/增肌）
Step 2: 问用户体重（kg）
Step 3: 问用户性别（男/女）
Step 4: 问用户活动量（久坐/轻度/中度/高度）

系统使用 Mifflin-St Jeor 方程计算：
  1. BMR（基础代谢率）
  2. TDEE（总每日能量消耗）
  3. 根据目标调整热量
  4. 分配三大营养素
```

**计算公式**：
```
BMR（男）= 10×体重(kg) + 6.25×身高(cm) - 5×年龄 + 5
BMR（女）= 10×体重(kg) + 6.25×身高(cm) - 5×年龄 - 161

TDEE = BMR × 活动系数
  久坐: 1.2
  轻度: 1.375
  中度: 1.55
  高度: 1.725

目标热量：
  减脂: TDEE × 0.8（20%赤字）
  维持: TDEE × 1.0
  增肌: TDEE × 1.15（15%盈余）

营养素分配：
  蛋白质: 体重 × 2.4 g/kg（减脂）或 1.8 g/kg（其他）
  脂肪: 体重 × 0.66 g/kg（减脂）或 1.1 g/kg（其他）
  碳水: (目标热量 - 蛋白质×4 - 脂肪×9) / 4
```

**用户场景**：
```
基米吃饭：你好！我是基米，帮你搭配食堂餐食 🍚

第一步：你的目标是什么？
  1. 减脂
  2. 维持体重
  3. 增肌
  > 3

第二步：你的体重是多少（kg）？
  > 70

第三步：你的性别是？
  1. 男
  2. 女
  > 1

第四步：你的活动量是？
  1. 久坐不动
  2. 轻度活动（每周1-3次）
  3. 中度活动（每周3-5次）
  4. 高度活动（每周6-7次）
  > 3

[系统自动计算...]

基米吃饭：根据你的信息，建议每天摄入：
  - 蛋白质：150g
  - 脂肪：60g
  - 碳水：200g

是否使用这个目标？(y/n)
```

#### F3：食堂菜品推荐

**是什么**：根据用户的营养目标，从食堂菜单中推荐具体的菜品组合。

**为什么需要**：
- 用户知道了自己的营养目标，但不知道食堂里吃什么能达到目标
- 食堂菜品多，用户选择困难
- 系统自动匹配，省时省力

**怎么工作**：
```
输入：用户的营养目标（蛋白质、脂肪、碳水）
      + 用户的偏好设置（忌口、餐次）
      + 食堂菜品数据库

处理：
  1. RAG检索：根据营养目标检索匹配的菜品
  2. LLM推理：生成推荐方案
  3. 营养计算：计算推荐方案的总营养

输出：推荐的菜品列表（含食堂、窗口、营养信息）
```

**用户场景**：
```
基米吃饭：正在为你推荐菜品...

推荐方案：

早餐：
  - 桃园一楼2号窗口：鸡蛋灌饼 ×1（蛋白质12g，脂肪8g，碳水25g）
  - 桃园一楼5号窗口：豆浆 ×1（蛋白质6g，脂肪2g，碳水8g）

午餐：
  - 橘园二楼1号窗口：清炒西兰花 ×1份（蛋白质5g，脂肪3g，碳水8g）
  - 橘园二楼3号窗口：红烧鸡腿 ×1份（蛋白质28g，脂肪12g，碳水5g）
  - 橘园一楼主食窗口：米饭 ×1份（蛋白质4g，脂肪0g，碳水60g）

晚餐：
  - 李园一楼1号窗口：番茄炒蛋 ×1份（蛋白质10g，脂肪8g，碳水6g）
  - 李园一楼2号窗口：清炒时蔬 ×1份（蛋白质3g，脂肪2g，碳水5g）
  - 李园一楼主食窗口：杂粮饭 ×1份（蛋白质5g，脂肪1g，碳水55g）
```

#### F4：一日三餐分配

**是什么**：将用户一天的总营养目标分配到早餐、午餐、晚餐（和加餐）。

**为什么需要**：
- 用户一天不是只吃一顿，需要分配到三餐
- 不同餐次的营养需求不同（早餐吃少、午餐吃饱、晚餐吃好）
- 帮助用户合理安排每餐吃什么

**怎么工作**：
```
输入：总营养目标（蛋白质150g，脂肪60g，碳水200g）

处理：按比例分配到各餐
  早餐：20%（蛋白质30g，脂肪12g，碳水50g）
  午餐：40%（蛋白质60g，脂肪24g，碳水80g）
  晚餐：30%（蛋白质45g，脂肪18g，碳水60g）
  加餐：10%（蛋白质15g，脂肪6g，碳水10g）

输出：每餐的营养目标
```

**分配比例**：
```
┌────────┬────────┬────────┬────────┐
│        │ 蛋白质 │ 脂肪   │ 碳水   │
├────────┼────────┼────────┼────────┤
│ 早餐   │ 30g    │ 12g    │ 50g    │
│ 午餐   │ 60g    │ 24g    │ 80g    │
│ 晚餐   │ 45g    │ 18g    │ 60g    │
│ 加餐   │ 15g    │ 6g     │ 10g    │
└────────┴────────┴────────┴────────┘

注：比例可根据用户习惯调整
```

**用户场景**：
```
基米吃饭：你的总营养目标已分配到三餐：

  早餐：蛋白质30g，脂肪12g，碳水50g
  午餐：蛋白质60g，脂肪24g，碳水80g
  晚餐：蛋白质45g，脂肪18g，碳水60g

是否按此分配推荐菜品？(y/n)
```

#### F5a：份量调整

**是什么**：用户可以调整某道菜的食用份量（如只吃2/3份）。

**为什么需要**：
- 食堂打菜是按份算的，但用户可能吃不完
- 用户可能只想吃一部分（如减肥时少吃点主食）
- 调整份量后，营养数据需要重新计算

**怎么工作**：
```
输入：用户指定某道菜的份量比例（如2/3）

处理：
  1. 找到用户指定的菜品
  2. 按比例计算营养成分
    原始：蛋白质28g，脂肪12g，碳水5g
    2/3份：蛋白质19g，脂肪8g，碳水3g
  3. 重新计算总营养

输出：调整后的菜品和营养数据
```

**用户场景**：
```
基米吃饭：推荐方案中：
  - 红烧鸡腿 ×1份（蛋白质28g，脂肪12g，碳水5g）

用户：这个我只吃2/3份

基米吃饭：已调整：
  - 红烧鸡腿 ×2/3份（蛋白质19g，脂肪8g，碳水3g）

[重新计算营养对比表]
```

**技术实现**：
```python
def adjust_portion(dish, portion_ratio):
    """
    调整菜品份量
    
    参数：
        dish: 菜品信息 {"name": "红烧鸡腿", "protein": 28, "fat": 12, "carbs": 5}
        portion_ratio: 份量比例（如 2/3 = 0.67）
    
    返回：
        调整后的菜品信息
    """
    adjusted = {
        "name": dish["name"],
        "portion": f"{portion_ratio}份",
        "protein": round(dish["protein"] * portion_ratio),
        "fat": round(dish["fat"] * portion_ratio),
        "carbs": round(dish["carbs"] * portion_ratio)
    }
    return adjusted
```

#### F5b：食材排除

**是什么**：用户可以排除某道菜中的特定食材（如只吃肉丝不吃青椒）。

**为什么需要**：
- 食堂菜品是固定的，但用户可能不喜欢其中某种食材
- 用户可能只想吃其中一部分（如青椒肉丝只吃肉）
- 排除后，营养数据需要重新计算

**怎么工作**：
```
输入：用户指定排除的食材（如"青椒"）

处理：
  1. 找到用户指定的菜品
  2. 根据食材比例，计算排除后的营养成分
    原始：蛋白质20g，脂肪10g，碳水8g
    排除青椒：蛋白质16g，脂肪8g，碳水2g
  3. 重新计算总营养

输出：调整后的菜品和营养数据
```

**用户场景**：
```
基米吃饭：推荐方案中：
  - 青椒肉丝 ×1份（蛋白质20g，脂肪10g，碳水8g）

用户：我不吃青椒，只吃肉丝

基米吃饭：已调整：
  - 青椒肉丝（只吃肉丝）（蛋白质16g，脂肪8g，碳水2g）

[重新计算营养对比表]
```

**技术实现**：
```python
def exclude_ingredient(dish, excluded_ingredient):
    """
    排除菜品中的特定食材
    
    参数：
        dish: 菜品信息
        excluded_ingredient: 要排除的食材名称
    
    返回：
        调整后的菜品信息
    """
    # 获取食材在菜品中的营养占比
    ingredient_ratio = get_ingredient_ratio(dish, excluded_ingredient)
    
    # 计算排除后的营养
    adjusted = {
        "name": dish["name"],
        "note": f"不含{excluded_ingredient}",
        "protein": round(dish["protein"] * (1 - ingredient_ratio["protein"])),
        "fat": round(dish["fat"] * (1 - ingredient_ratio["fat"])),
        "carbs": round(dish["carbs"] * (1 - ingredient_ratio["carbs"]))
    }
    return adjusted
```

#### F6：忌口设置

**是什么**：用户设置全局不吃的食材，系统推荐时自动排除包含这些食材的菜品。

**为什么需要**：
- 有些用户有食物过敏（如花生过敏）
- 有些用户有宗教饮食限制（如不吃猪肉）
- 有些用户就是不喜欢某种食材（如不吃香菜）
- 设置一次，后续推荐自动排除

**怎么工作**：
```
输入：用户选择忌口食材（如香菜、辣椒）

处理：
  1. 保存用户的忌口列表
  2. 推荐菜品时，过滤掉包含忌口食材的菜品
  3. 如果某道菜可以去掉忌口食材，使用F5b功能

输出：不包含忌口食材的推荐方案
```

**用户场景**：
```
基米吃饭：你有忌口吗？（可多选，输入编号用逗号分隔，没有输入0）
  1. 香菜
  2. 花生
  3. 海鲜
  4. 牛奶
  5. 辣椒
  6. 鸡蛋
  > 1,5

基米吃饭：已记录你的忌口：香菜、辣椒
后续推荐将避免包含这些食材的菜品。
```

#### F7：菜品替代

**是什么**：用户不喜欢推荐的某道菜，系统推荐营养相近的替代菜品。

**为什么需要**：
- 用户可能就是不想吃某道菜
- 用户可能吃腻了某道菜
- 需要快速找到营养相近的替代品

**怎么工作**：
```
输入：用户指定要替换的菜品

处理：
  1. 获取原菜品的营养数据
  2. 在数据库中检索营养相近的菜品
  3. 过滤掉用户忌口的菜品
  4. 返回替代方案

输出：替代菜品 + 营养变化对比
```

**用户场景**：
```
基米吃饭：推荐方案中：
  - 红烧鸡腿 ×1份（蛋白质28g，脂肪12g，碳水5g）

用户：我不想吃这个，换一个

基米吃饭：已替换为：
  - 清蒸鱼 ×1份（蛋白质25g，脂肪5g，碳水2g）

营养变化：蛋白质-3g，脂肪-7g，碳水-3g
```

**技术实现**：
```python
def find_alternative(original_dish, food_database, blocked_ingredients):
    """
    查找营养相近的替代菜品
    
    参数：
        original_dish: 原菜品信息
        food_database: 食堂菜品数据库
        blocked_ingredients: 用户忌口列表
    
    返回：
        替代菜品列表
    """
    # 计算原菜品的营养向量
    original_vector = [original_dish["protein"], original_dish["fat"], original_dish["carbs"]]
    
    # 检索营养相近的菜品
    alternatives = []
    for dish in food_database:
        if dish["name"] == original_dish["name"]:
            continue
        if any(ing in blocked_ingredients for ing in dish["ingredients"]):
            continue
        
        dish_vector = [dish["protein"], dish["fat"], dish["carbs"]]
        similarity = calculate_similarity(original_vector, dish_vector)
        alternatives.append({"dish": dish, "similarity": similarity})
    
    # 按相似度排序
    alternatives.sort(key=lambda x: x["similarity"], reverse=True)
    
    return alternatives[:3]  # 返回前3个最相似的
```

#### F8：喜好学习

**是什么**：系统根据用户的历史选择，优化后续推荐。

**为什么需要**：
- 每个人口味不同，通用推荐可能不适合所有人
- 用户经常选择的菜品，应该优先推荐
- 用户经常跳过的菜品，应该降低优先级
- 越用越精准，提升用户体验

**怎么工作**：
```
输入：用户的历史行为
  - 选择了哪些菜品
  - 跳过了哪些菜品
  - 替换了哪些菜品
  - 评分了哪些菜品

处理：
  1. 统计用户偏好
    - 经常选择的菜品 → 权重+1
    - 经常跳过的菜品 → 权重-1
    - 替换过的菜品 → 权重-0.5
  2. 生成用户画像
    - 口味偏好（辣/不辣、荤/素）
    - 食堂偏好（常去哪个食堂）
    - 价格偏好（预算范围）
```

#### F9：营养对比表

**是什么**：将用户的目标和实际推荐结果放在一起对比，让用户一眼看出"差多少"。

**为什么需要**：
- 用户不知道推荐的够不够
- 用户不知道差多少
- 用户不知道超了多少
- 对比表让用户一目了然，决定是否需要调整

**怎么工作**：
```
输入：用户目标 + 推荐方案的营养总和

处理：
  1. 计算差距 = 推荐总计 - 目标
  2. 计算达成率 = 推荐总计 / 目标 × 100%
  3. 生成调整建议

输出：对比表 + 建议
```

**对比表格式**：
```
┌────────────┬──────────┬──────────┬──────────┐
│            │ 蛋白质   │ 脂肪     │ 碳水     │
├────────────┼──────────┼──────────┼──────────┤
│ 目标       │ 150g     │ 60g      │ 200g     │
│ 推荐总计   │ 145g     │ 55g      │ 195g     │
│ 差距       │ -5g      │ -5g      │ -5g      │
│ 达成率     │ 97%      │ 92%      │ 98%      │
├────────────┼──────────┼──────────┼──────────┤
│ ⚠️ 建议    │ 差距较小，可以直接采纳           │
└────────────┴──────────┴──────────┴──────────┘
```

```python
def calculate_nutrition_comparison(target, recommended):
    """
    计算营养对比表
    
    参数：
        target: 用户目标 {"protein": 150, "fat": 60, "carbs": 200}
        recommended: 推荐总计 {"protein": 145, "fat": 55, "carbs": 195}
    
    返回：
        对比结果
    """
    result = {}
    
    for nutrient in ["protein", "fat", "carbs"]:
        target_val = target[nutrient]
        recommended_val = recommended[nutrient]
        
        # 差距
        diff = recommended_val - target_val
        
        # 达成率
        achievement_rate = (recommended_val / target_val) * 100
        
        result[nutrient] = {
            "target": target_val,
            "recommended": recommended_val,
            "diff": diff,
            "achievement_rate": round(achievement_rate, 1)
        }
    
    return result


def generate_suggestions(comparison):
    """
    根据对比结果生成建议
    """
    suggestions = []
    
    # 蛋白质不足
    if comparison["protein"]["diff"] < -20:
        suggestions.append("蛋白质严重不足，建议补充高蛋白食物：鸡蛋、鸡胸肉、牛奶")
    elif comparison["protein"]["diff"] < -5:
        suggestions.append("蛋白质略有不足，可以加1-2个鸡蛋")
    
    # 脂肪超标
    if comparison["fat"]["diff"] > 10:
        suggestions.append("脂肪超标，建议减少油炸食品，选择清蒸、水煮方式")
    
    # 碳水不足
    if comparison["carbs"]["diff"] < -30:
        suggestions.append("碳水不足，建议增加主食份量")
    
    return suggestions
```

## 七、风险与应对

## 八、总结

### 8.1 核心策略

```
功能固定 + 数据扩展

功能：一次性开发完成（F1-F11）
数据：分批扩展（MVP→1个食堂→片区→全校）
```

### 8.2 版本路径

```
MVP（公开数据）→ V1.0（1个食堂）→ V1.5（片区）→ V2.0（全校）
```

### 8.3 关键成功因素

1. **数据质量**：食堂菜品数据的准确性和完整性
2. **模型效果**：LLM + RAG 的推荐质量
3. **用户体验**：交互流程是否顺畅
4. **持续更新**：菜品数据的及时更新
